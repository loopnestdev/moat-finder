# CLAUDE.md — moat-finder/backend

Read docs/ARCHITECTURE.md, docs/DATABASE.md, and docs/FEATURES.md before
making any changes to the backend.

---

## Stack

- Node.js v20 LTS
- Express 4 + TypeScript strict
- Supabase JS client v2 (server-side)
- Anthropic SDK (official `@anthropic-ai/sdk`)
- Zod — input validation on all routes
- Helmet — security headers
- express-rate-limit — rate limiting
- cors — restricted to frontend origin only

## Key Rules

1. **Never use `req.ip`** — always use `req.headers['cf-connecting-ip']` via
   `utils/ip.ts`. Fall back to `req.socket.remoteAddress` in local dev only.

2. **Never bypass RLS** — use the Supabase anon key for user-facing queries.
   Use the service role key ONLY for admin operations and audit log writes.

3. **All routes use Zod validation** — validate ticker symbol and all body params
   before any database or pipeline operation.

4. **Pipeline is always sequential** — never parallelise steps. Each step's
   output is passed as context to the next. See docs/ARCHITECTURE.md Step map.

5. **SSE for streaming** — research and update endpoints stream progress via
   Server-Sent Events. Set headers: `Content-Type: text/event-stream`,
   `Cache-Control: no-cache`, `Connection: keep-alive`.

6. **Audit every action** — call the audit middleware for every meaningful action.
   See `audit_log.action` enum in docs/DATABASE.md.

7. **Error responses** — always return `{ error: string, code: string }` shape.
   Never expose stack traces or internal errors to the client.

## Testing Rules

- Test file per service: `pipeline.test.ts`, `diff.test.ts`, `auth.test.ts`
- Mock Anthropic SDK and Supabase client in tests — never call live APIs in tests
- Every Zod schema has a test for valid and invalid inputs
- Every util function has 100% branch coverage
