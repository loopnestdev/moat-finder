## Key Rules

1. **Never use `req.ip`** — always use `utils/ip.ts` which reads `CF-Connecting-IP`. Fall back to `req.socket.remoteAddress` in local dev only.

2. **Never bypass RLS** — use `anonClient` for user-facing queries. Use `adminClient` (service role) ONLY for admin operations, audit log writes, and pipeline saves.

3. **All routes use Zod validation** — validate ticker symbol and all body params before any database or pipeline operation.

4. **Pipeline execution order**:
   - Step 1 (Discovery) always runs first and completes before anything else.
   - Steps 2-6 run **concurrently** via `Promise.allSettled`.
   - Step 7 (Synthesis) runs last, receiving all step outputs.
   - `runUpdatePipeline()` (smart update) checks `existingReport.management_rating`:
     - **Missing**: Step 2 re-runs fresh concurrently with Steps 3/5/6. Step 4 is always reused from cache.
     - **Present**: Step 2 reconstructed from cache (with `management_rating` carried through). Only Steps 3/5/6 run fresh.
   - `runPipeline()` invalidates any cached Step 2 checkpoint missing `management_rating` (deletes from map + DB), forcing a fresh re-run.

5. **SSE for streaming** — set headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`. `EmitFn` writes `data: <json>\n\n`. Step status values: `complete`, `error`, `cached`.

6. **Prompt caching** — Steps 2-6 pass Step 1 output as a cached content block (`cache_control: { type: 'ephemeral' }`). This caches the ~600-token context across parallel calls.

7. **Step 4 (Risk Red Team)** — must be capped to 4 web searches maximum and < 800 words. Uncapped, it can take 600+ seconds.

8. **Audit every action** — call audit middleware for every meaningful action (`research_triggered`, `research_updated`, `report_viewed`, etc.).

9. **Error responses** — always return `{ error: string, code: string }`. Never expose stack traces.

10. **Score calculation** — Step 7 uses a weighted rubric (sector momentum 20%, growth velocity 25%, valuation vs peers 20%, moat quality 20%, execution risk 15%). A company with >100% YoY growth in a hot sector must score ≥ 5.0. `management_rating` is NOT a scoring factor — it is injected into `parsed.report` after Step 7 synthesis so the scoring LLM never sees it.

## Pipeline Types (report.types.ts)

- `SSEEvent.status`: `'complete' | 'error' | 'cached'`
- `ReportJson.quarterly_results?: QuarterlyResult[]` — optional, absent in old reports
- `ReportJson.risk_factors?: string[]` — always default to `[]` when reading old reports
- `ReportJson.management_rating?: ManagementRating` — optional, absent in pre-v0.5.2 reports; injected post-synthesis in `runStep7()`
- `Step2Output.management_rating?: ManagementRating` — generated by Step 2 Deep Dive prompt; carried into `parsed.report` after LLM synthesis completes
- `Step3Output.quarterly_results?: QuarterlyResult[]` — carried to final report in Step 7
- `runPipeline(ticker, emit)` — full 7-step new research; invalidates stale Step 2 checkpoints missing `management_rating`
- `runUpdatePipeline(ticker, existingReport, emit)` — smart update; re-runs Step 2 if `management_rating` absent, otherwise carries it through from `existingReport`
- `deleteStepCheckpoint(ticker, runId, stepNumber)` — deletes a single checkpoint row (used to force step re-run on resume)
