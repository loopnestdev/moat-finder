## Key Rules

1. **Never use `req.ip`** ‚Äî always use `utils/ip.ts` which reads `CF-Connecting-IP`. Fall back to `req.socket.remoteAddress` in local dev only.

2. **Never bypass RLS** ‚Äî use `anonClient` for user-facing queries. Use `adminClient` (service role) ONLY for admin operations, audit log writes, and pipeline saves.

3. **All routes use Zod validation** ‚Äî validate ticker symbol and all body params before any database or pipeline operation.

4. **Pipeline execution order**:
   - Step 1 (Discovery) always runs first and completes before anything else.
   - Steps 2-6 run **concurrently** via `Promise.allSettled`.
   - Step 7 (Synthesis) runs last, receiving all step outputs.
   - `runUpdatePipeline()` (smart update) checks `existingReport.management_rating`:
     - **Missing**: Step 2 re-runs fresh concurrently with Steps 3/5/6. Step 4 is always reused from cache.
     - **Present**: Step 2 reconstructed from cache (with `management_rating` carried through). Only Steps 3/5/6 run fresh.
   - `runPipeline()` invalidates any cached Step 2 checkpoint missing `management_rating` (deletes from map + DB), forcing a fresh re-run.

5. **SSE for streaming** ‚Äî set headers: `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`. `EmitFn` writes `data: <json>\n\n`. Step status values: `complete`, `error`, `cached`.

6. **Prompt caching** ‚Äî Steps 2-6 pass Step 1 output as a cached content block (`cache_control: { type: 'ephemeral' }`). This caches the ~600-token context across parallel calls.

7. **Step 4 (Risk Red Team)** ‚Äî must be capped to 4 web searches maximum and < 800 words. Uncapped, it can take 600+ seconds.

8. **Audit every action** ‚Äî call audit middleware for every meaningful action (`research_triggered`, `research_updated`, `report_viewed`, etc.).

9. **Error responses** ‚Äî always return `{ error: string, code: string }`. Never expose stack traces.

10. **Score calculation** ‚Äî Step 7 uses a weighted rubric (sector momentum 20%, growth velocity 25%, valuation vs peers 20%, moat quality 20%, execution risk 15%). A company with >100% YoY growth in a hot sector must score ‚â• 5.0. `management_rating` is NOT a scoring factor ‚Äî it is injected into `parsed.report` after Step 7 synthesis so the scoring LLM never sees it.

11. **Competitor/peer selection must match revenue-driving product line** (since v0.8.6) ‚Äî Step 1 (`competitors`) and Step 3 (`valuation_table` peers) prompts both require the model to match companies by the SAME specific product/service line that generates the MAJORITY of the subject company's revenue, not merely a shared broad sector or theme. Foreign-listed tickers (Korean, Japanese, Taiwanese, etc.) must be actively considered when they are the true industry leaders ‚Äî do not let the model default to familiar US-listed names. Step 3 additionally requires each scenario's `comp_multiple` to exactly equal that peer's `ps_ratio` in `valuation_table` (needed for the frontend's Napkin Math comp selector, `frontend/src/lib/napkinMath.ts`, to reliably extrapolate a price for any peer).

## Pipeline Types (report.types.ts)

- `SSEEvent.status`: `'complete' | 'error' | 'cached'`
- `ReportJson.quarterly_results?: QuarterlyResult[]` ‚Äî optional, absent in old reports
- `ReportJson.risk_factors?: string[]` ‚Äî always default to `[]` when reading old reports
- `ReportJson.management_rating?: ManagementRating` ‚Äî optional, absent in pre-v0.5.2 reports; injected post-synthesis in `runStep7()`
- `Step2Output.management_rating?: ManagementRating` ‚Äî generated by Step 2 Deep Dive prompt; carried into `parsed.report` after LLM synthesis completes
- `ReportJson.technological_advantage?: string` ‚Äî optional, absent in pre-v0.8.7 reports; plain-English tech explanation generated by Step 2, injected post-synthesis in `runStep7()` (same pattern as `management_rating`) and carried through cached-Step-2 reconstruction in `runUpdatePipeline()`
- `Step3Output.quarterly_results?: QuarterlyResult[]` ‚Äî carried to final report in Step 7
- `runPipeline(ticker, emit)` ‚Äî full 7-step new research; invalidates stale Step 2 checkpoints missing `management_rating`
- `runUpdatePipeline(ticker, existingReport, emit)` ‚Äî smart update; re-runs Step 2 if `management_rating` absent, otherwise carries it through from `existingReport`
- `deleteStepCheckpoint(ticker, runId, stepNumber)` ‚Äî deletes a single checkpoint row (used to force step re-run on resume)

## Testing Rules

- One test file per service: `pipeline.test.ts`, `diff.test.ts`, `saveResearch.test.ts`
- Mock Anthropic SDK and Supabase client ‚Äî never call live APIs in tests. `SUPABASE_URL` in `vitest.config.ts` is a fake host (`test.supabase.co`) ‚Äî any unmocked Supabase call will hit real DNS and hang/timeout, it will not fail fast.
- Any test touching `pipeline.ts`'s `runPipeline()` must also mock `services/confirmation.ts` (`registerConfirmation` ‚Üí resolve `{ confirmed: true }` immediately) ‚Äî otherwise the post-Discovery confirmation gate (added v0.7.1) stalls the test for up to 60s.
- `vi.mock(...)` factories are hoisted above the file ‚Äî a plain `const mockFn = vi.fn()` referenced inside one throws a TDZ `ReferenceError` under vitest 3.2.x. Use `vi.hoisted(() => vi.fn())` instead.
- Every Zod schema tested for valid and invalid inputs
- Every util function at 100% branch coverage
